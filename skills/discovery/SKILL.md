---
name: discovery
description: 当用户提出新需求、新问题、新想法或表达困惑时使用。通过提问和拆解，输出结构化的理解。
---

## 状态机

```
LISTEN ──[gate:ready]──→ EXPLORE ──[gate:sufficient]──→ STRUCTURE ──[gate:valid]──→ CONFIRM ──→ OUTPUT ──→ END
  ↑                        ↑  ↓                           ↑                          │
  └── 概念不清时回退 ──────┘  └── 循环直到充分 ────────────┘                          │
                                                            ↑                          │
                                                            └── 用户要求修改时回退 ────┘
```

## 边界

Discovery 回答"做什么"，Planning 回答"怎么做"。如果你发现自己在回答"怎么做"，你已经越界了。

**不做**：技术选型、架构设计、输出代码/伪代码、展示"方案"让用户选。这些全是 Planning 的职责。

**不跳过**：哪怕需求看起来很明确，也必须走完 LISTEN → CONFIRM。你以为明确 ≠ 真的明确。返工成本远大于多问一轮。

**不批量提问**：一次只问一个问题，用 AskUserQuestion 工具。用户一次处理一个问题质量最高。

**铁律**：没有用户确认的需求理解，不进入 Planning。

## STATE: LISTEN

接收用户表达，澄清概念，识别意图类型。

### 动作

1. **完整接收**：不打断，不急于给方案，先听完用户说什么。

2. **概念澄清**：对用户表达中的每个关键概念、术语、引用——
   - 不确定它是什么 → 先去查（读文件、搜索、问用户）
   - 不假设你知道，不用自己的理解替代事实
   - 项目中已有的文件、配置、约定，必须先读再评论
   - 顺序：先问"是什么" → 再问"为什么" → 最后问"怎么办"

3. **识别类型**：判断用户输入属于哪种——
   - **需求**：想要实现某个目标 → 后续拆解为步骤
   - **问题**：想要理解某个现象 → 后续拆解为层次
   - **想法**：想要探索某个方向 → 后续拆解为论点
   - **困惑**：想要解决某个疑惑 → 后续拆解为已知/未知

### 转换条件 [gate:ready]

| 检查项 | 通过条件 |
|--------|---------|
| 倾听 | 已完整接收用户表达，未打断、未急于给方案 |
| 概念澄清 | 关键概念已查证，不存在"我猜它是…"的情况 |
| 类型识别 | 已判定为 需求/问题/想法/困惑 之一 |

未通过 → 留在 LISTEN，回到对应动作。全部通过 → 进入 EXPLORE。

---

## STATE: EXPLORE

循环探索，每轮做一件事，做完检查是否可以退出。

### 子动作（每轮选一个执行）

#### 提问澄清

针对模糊点提出具体问题：
- **目标**：你想达成什么？
- **动机**：为什么想做这个？
- **约束**：有什么限制条件？
- **边界**：什么不在范围内？

提问纪律：一次只问一个问题，用 AskUserQuestion 工具，多选项优先。提出 2-3 个方向并说明各自取舍，让用户选择。先展示你的理解，用户确认后再进入下一步。

#### 深度搜索

对用户需求涉及的领域进行主动搜索调研。先用 WebSearch 广度搜索获取全貌，根据结果判断是否需要用 smart-fetch 深入抓取关键文章全文，信息不足则调整关键词再搜。

必搜维度（每个维度至少一组关键词）：
- **竞品/开源组件**：目标产品是否有官方开源项目、SDK 等可复用资源
- **同类项目的成熟实现**：GitHub 上 stars 较高的同类项目
- **领域坑点**：已知的技术限制、常见失败模式

注意：搜索给你领域知识，不给你用户意图。搜完必须回到提问澄清环节。

#### 加载领域 Skill

根据需求涉及的领域，加载相关外部 Skill（如 UI/UX Pro Max、vue-best-practices 等）。在认知拓展阶段就引入专业能力，让 Skill 的领域知识影响需求理解和方向判断。

#### 产品调性探索（涉及前端时必做）

当项目包含用户界面时，完成调性摸排：

1. **搜索同类产品**：搜索竞品 UI 截图、设计系统、Landing Page
2. **提炼调性描述**：用文字描述整体产品气质
3. **提取设计关键词标签集**（四个维度各 1-3 个关键词）：
   - 配色倾向：如 深色主题 / 中性灰 / 高饱和撞色
   - 字体风格：如 无衬线 / 等宽技术感 / 圆体亲和
   - 布局模式：如 侧边栏+画布 / 顶栏+卡片流 / 全屏沉浸
   - 交互风格：如 拖拽优先 / 表单驱动 / 对话式
4. **用户确认**：使用 AskUserQuestion 展示调性描述和关键词，让用户选择或修改

#### 探索性验证（按需）

当文字描述不足以做出判断时，可以通过子代理创建探索性制品：
- 风格对照页、原型草图（粗糙 HTML/CSS/JS）、流程验证脚本、API 数据验证

探索性制品存放在 `.claude/exploration/` 下，保留到 Verification 阶段完成。

### 转换条件 [gate:sufficient]

| 检查项 | 通过条件 |
|--------|---------|
| 用户意图 | 目标、动机、约束、边界都已确认 |
| 领域知识 | 至少覆盖：竞品、开源方案、领域坑点 |
| 产品调性 | 涉及前端时，调性已确认（不涉及前端则跳过） |
| 无残留疑问 | 没有剩余的"我不确定用户想要…" |

任一未通过 → 留在 EXPLORE，回到对应子动作。全部通过 → 进入 STRUCTURE。

---

## STATE: STRUCTURE

将理解整理为清晰的结构化输出。

### 结构选择

根据内容本身的自然形态选择——
- 树状：模块/功能分解
- 流状：用户动线/步骤序列
- 链状：论证/因果推理
- 图状：多组件互相依赖

不要强行用树状结构组织所有内容。流就是流，链就是链。

### 深度标准

每个最小可验证单元必须同时满足：
1. 单个子代理可独立完成
2. 完成标准无歧义——任何两个人看到描述，对"做完了"的理解一致
3. 不含未决选择——所有选择已在上层做完

未达标的单元继续拆解，直到全部达标。最小可验证单元描述的是用户需求和动线，不涉及代码实现细节。

### 粒度示例

| 粒度 | 示例 | 判定 |
|------|------|------|
| 太粗 | "剧本拆解模块" | ❌ 实现视角 |
| 太细 | "用 FastAPI 写 /api/parse 接口" | ❌ 侵入实现 |
| 正确 | "用户粘贴小说文本后，系统自动识别出所有角色及其特征描述" | ✅ 用户视角，无歧义 |

### 转换条件 [gate:valid]

| 检查项 | 通过条件 |
|--------|---------|
| 拆解粒度 | 每个单元都满足三个深度标准 |
| 无实现侵入 | 输出中不包含技术选型、架构设计、代码片段 |

未通过 → 回到 EXPLORE 或重新拆解。通过 → 进入 CONFIRM。

---

## STATE: CONFIRM

与用户确认结构化输出是否正确。如果用户要求修改，回退到 EXPLORE 或 STRUCTURE。

用户确认通过 → 进入 OUTPUT。

---

## STATE: OUTPUT

将结构化理解通过子代理写入 PRD 文件。

PRD 文件存放在项目目录下 `.claude/discovery/PRD.md`，内容包括：
- 所有最小可验证单元（按自然结构组织）
- 用户动线/流程（如适用）
- 探索性验证的结论引用（如有）
- 风格决策和调性关键词（涉及前端时）
- 验收标准
- 边界条件和排除项

PRD 是后续所有阶段的工作依据。Planning 读取 PRD 进行架构设计，Execution 按 PRD 验收。

### 阶段结束

用 AskUserQuestion 向用户确认：
选项必须包含：
- "进入 Planning"（推荐）
- "需要补充探索"
