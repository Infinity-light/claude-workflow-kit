## 铁律

没有用户确认的需求理解，不进入 Planning。

## 常见借口（都是错的）

| 你可能会想 | 为什么是错的 |
|-----------|------------|
| "需求很明确不需要探索" | 你以为明确不代表真的明确，用户脑中的画面和你理解的可能完全不同 |
| "先做着看，做完再调整" | 返工成本远大于多问一轮的成本 |
| "用户会觉得我问太多" | 用户更怕你做错方向。一次问一个问题，不会烦 |
| "我搜过了，已经够了解了" | 搜索给你的是领域知识，不是用户意图。两者不能替代 |

# 发现阶段

## 识别信号

- 任何新对话或新主题的开始
- 用户提出需求、问题、想法或困惑
- 当前讨论进入新的方向
- 感觉需要"停下来理解"的时刻

## 核心流程

### 1. 倾听
不要急于给方案，先完整接收用户的表达。

### 2. 识别类型
判断用户输入的类型：
- **需求**：想要实现某个目标 → 拆解为步骤
- **问题**：想要理解某个现象 → 拆解为层次
- **想法**：想要探索某个方向 → 拆解为论点
- **困惑**：想要解决某个疑惑 → 拆解为已知/未知

### 3. 提问澄清
针对模糊点提出具体问题：
- **目标**：你想达成什么？
- **动机**：为什么想做这个？
- **约束**：有什么限制条件？
- **边界**：什么不在范围内？

### 3.5 提问纪律（Brainstorming）

- **一次只问一个问题**，用 AskUserQuestion 工具，多选项优先
- 提出 2-3 个方向并说明各自取舍，让用户选择
- 不要一次性抛出所有问题的清单
- 先在聊天中展示你的理解，用户确认后再进入下一步

### 4. 加载领域 Skill
根据用户需求涉及的领域，积极探索并加载相关的外部 Skill（如 UI/UX Pro Max、vue-best-practices 等）。在认知拓展阶段就引入专业能力，让 Skill 的领域知识影响需求理解和方向判断，而不是等到 planning 才加载。

### 5. 深度搜索
在形成结论前，对用户需求涉及的领域进行主动搜索调研。采用动态循环策略：先用 WebSearch 广度搜索多组关键词获取全貌，根据结果判断是否需要用 smart-fetch 深入抓取关键文章全文，如果信息仍不足则调整关键词再搜，直到掌握足够信息做出判断。搜索的目的是回答"这个领域别人是怎么做的、有哪些成熟方案、有哪些坑"，避免闭门造车。

**必搜维度**（每个维度至少一组关键词）：
- **目标产品/竞品的开源组件**：搜索目标产品是否有官方开源项目、SDK、画布系统等可直接复用的资源（例：搜"Coze open source"、"扣子 开源 canvas"）
- **同类项目的成熟实现**：GitHub 上 stars 较高的同类项目，作为架构参考（例：搜"workflow builder open source github"）
- **技术栈的最新实践**：所选技术栈的最新版本、已知坑点、社区推荐方案

### 6. 产品调性探索（涉及前端时必做）

当项目包含用户界面时，在结构化输出之前完成调性摸排：

1. **搜索同类产品**：搜索竞品、同类产品的 UI 截图、设计系统、Landing Page，收集参考链接
2. **提炼调性描述**：用文字描述整体产品气质（如"极简专业、信息密度高、克制用色"或"活泼年轻、圆角卡片、插画风"）
3. **提取设计关键词标签集**：从以下四个维度各给出 1-3 个关键词：
   - **配色倾向**：如 深色主题 / 中性灰 / 高饱和撞色
   - **字体风格**：如 无衬线 / 等宽技术感 / 圆体亲和
   - **布局模式**：如 侧边栏+画布 / 顶栏+卡片流 / 全屏沉浸
   - **交互风格**：如 拖拽优先 / 表单驱动 / 对话式
4. **用户确认**：使用 AskUserQuestion 展示调性描述和关键词，让用户选择或修改

### 7. 结构化输出
将理解和搜索发现整理为清晰的结构（步骤清单、问题分解、论点框架等）。

### 8. 确认
与用户确认理解是否正确。

## 输出

完成发现阶段后，输出：
- 结构化的理解（步骤清单、问题分解、或论点框架）
- 已确认的目标和约束
- **产品调性**（涉及前端时）：文字描述 + 参考链接 + 四维关键词标签集
- 建议进入的下一阶段（通常是 planning）

## 阶段结束

当理解已确认，用 AskUserQuestion 向用户确认下一步：

选项必须包含：
- "进入 Planning"（推荐）
- "需要补充探索"
- "暂停"
