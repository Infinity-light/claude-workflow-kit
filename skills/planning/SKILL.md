## 铁律

没有自包含的模块文档不进入 Execution。产出者和验证者必须是不同的子代理。

## 常见借口（都是错的）

| 你可能会想 | 为什么是错的 |
|-----------|------------|
| "任务太简单不需要走流程" | 简单任务走流程成本极低，不走流程出错成本极高 |
| "我自己检查一下就行" | 自己批改自己的作业永远觉得没问题。产出和验证必须分离 |
| "模块文档不需要写任务序列" | 没有任务序列的文档，执行者只能猜。猜就会出错 |
| "先开始做，边做边补文档" | 边做边补 = 没有文档。执行者需要在开始前看到完整路径 |

## 核心原则

- **主Agent 只调度，不产出具体文档内容** — 具体产出交给子代理
- **产出-验证分离** — 每个产出物由一个子代理写，另一个子代理对照检查清单验证
- **验证用检查清单，不用主观评分** — "有没有"比"好不好"可靠
- **规划阶段不写代码文件** — 用文字描述到伪代码/配置级别
- **未经用户确认不得写入任何文件**

## 核心流程

```
步骤1: 分析需求 + 技术栈选型（主Agent）
  ↓
步骤2: 前端设计（子代理产出 + 子代理验证 + 用户确认）
  ↓
步骤3: 架构总览文档（子代理产出 + 子代理验证 + 用户确认）
  ↓
步骤4: 逐模块设计文档（每个模块：子代理产出 + 子代理验证）
  ↓
步骤5: 用户确认 → 固化（写入文件 + TaskCreate）
```

---

### 步骤1: 分析需求 + 技术栈选型

**执行者**：主Agent 自己做。

1. **回顾 discovery 产出**：核心目标、约束条件、成功标准、产品调性（如有）
2. **探索方案**：查询依赖文档（优先 context7，备用 WebSearch）
3. **技术栈选型**（涉及前端时必做）：
   - 列出决策点
   - 逐个研究 2-3 个候选方案
   - **每个决策点单独用 AskUserQuestion 让用户选择**
   - 记录所有选型结论
4. **全链路约束探测**：按需检查本地环境、密钥、远程服务器、DNS、Git 等，有阻塞项立即告知用户

**不需要子代理验证** — 这步的产出是用户的选择和环境确认。

---

### 步骤2: 前端设计（涉及前端时必做）

**执行者**：子代理A（产出）+ 子代理B（验证）

1. **启动产出子代理**：
   - 传入：discovery 的产品调性 + 步骤1的技术选型结论
   - 指令：调用 `ui-ux-pro-max` Skill，为每个页面产出设计方案
   - 产出要求：布局描述、组件层级树、核心交互流程

2. **启动验证子代理**：
   - 传入：产出子代理的结果
   - 指令：读取 `references/frontend-design-checklist.md`，逐项验证
   - 不通过 → 列出缺失项 → 启动新子代理修补 → 再验证

3. **主Agent 用 AskUserQuestion 逐页让用户确认设计**

**产出物暂存**：前端设计方案保留在主Agent 上下文中，后续注入模块文档。

---

### 步骤3: 架构总览文档

**执行者**：子代理A（产出）+ 子代理B（验证）

1. **启动产出子代理**：
   - 传入：步骤1的技术栈选型 + 步骤2的前端设计 + discovery 产出
   - 指令：按以下结构编写架构总览文档：
     - 项目概述（名称、目标、技术栈汇总）
     - 目录结构
     - 环境变量完整清单（变量名、用途、开发/生产默认值）
     - 前后端通信约定（API base URL 配置、错误响应格式、认证 token 传递）
     - 数据库设计（Prisma schema 格式，含关系和枚举）
     - 模块列表 + 依赖关系 + 开发顺序

2. **启动验证子代理**：
   - 传入：产出子代理的结果
   - 指令：读取 `references/architecture-checklist.md`，逐项验证
   - 不通过 → 修补 → 再验证

3. **主Agent 向用户展示架构文档，用 AskUserQuestion 确认**

**产出物**：`architecture.md`，待步骤5写入 `.claude/plan/`

---

### 步骤4: 逐模块设计文档（核心步骤）

**按模块依赖顺序，逐个处理。** 每个模块：

1. **启动产出子代理**：
   - 传入：`references/module-template.md`（强制模板）+ 架构总览文档 + 该模块对应的前端设计
   - 指令：按模板编写该模块的完整自包含文档（含设计 + API规格 + 任务序列）
   - **关键**：文档第5节"任务序列"必须包含在文档内部

2. **启动验证子代理**：
   - 传入：产出子代理的结果 + `references/module-checklist.md`
   - 指令：逐项验证，输出"通过/未通过 + 缺失项列表"
   - 不通过 → 新子代理修补 → 新子代理再验证

**产出物**：每个模块一个文件（如 `listening.md`），待步骤6写入 `.claude/plan/`

---

### 步骤4.5: 外部资源汇总与验证

**执行者**：主Agent 汇总 + 子代理验证

1. **主Agent 汇总**：从所有模块文档的第5节"外部资源依赖"中提取资源声明，去重合并
2. **启动验证子代理**：
   - 对每个资源：搜索免费/开源替代方案、检查 API Key 可用性（调用 key-reader Skill）、验证资源可获取性
   - 数据源类资源：确认获取方式（爬取/下载/API/手动导入）
   - 输出：资源可用性报告（可用/需采购/需用户提供）
3. **主Agent 用 AskUserQuestion 向用户确认资源清单**，不可用的资源需用户决定替代方案
4. **将确认后的资源清单追加到架构总览文档的"外部资源"章节**

---

### 步骤5: 固化（原步骤5）

1. **用 AskUserQuestion 让用户确认全部文档**
2. **写入文件**：所有文档写入项目目录 `.claude/plan/`
3. **TaskCreate 创建任务链**：
   - 每个模块一个 Task
   - description：`读取 .claude/plan/{模块名}.md，按文档第5节"任务序列"逐个执行，每完成一个任务运行其验证命令确认通过`
   - blockedBy：按模块依赖顺序设置

---

## 阶段结束

固化完成后，用 AskUserQuestion 向用户确认下一步：

选项必须包含：
- "进入 Execution"（推荐）
- "需要修改计划"
- "暂停"
