## 铁律

没有微任务清单不进入 Execution。方案必须细到"照着做就行"。

## 常见借口（都是错的）

| 你可能会想 | 为什么是错的 |
|-----------|------------|
| "任务太简单不需要拆微任务" | 简单任务拆完也就 1-2 个微任务，成本极低，但能防止遗漏 |
| "模块拆分已经够细了" | 模块是架构单元，微任务是执行单元。模块内部还需要拆成可独立验证的步骤 |
| "先开始做，边做边拆" | 边做边拆 = 没有拆。执行者需要在开始前就看到完整路径 |
| "微任务描述不需要写验证命令" | 没有验证命令的任务无法判断是否完成，等于没有交付标准 |

# 规划阶段

## 核心原则

**方案必须细到"照着做就行"。** 用户不应该看完方案后还需要再想"具体怎么操作"。

- 规划阶段不写代码文件，但用文字描述到伪代码/配置级别
- 默认输出最细粒度，不做概述
- 聚焦"做什么"和"怎么做"，不做时间分配（除非用户要求）
- 未经用户确认不得写入任何文件

## 识别信号

- 用户问"怎么做"、"如何实现"
- 需求已清晰，但缺少执行方案
- 需要在多个方案中做选择

## 核心流程

### 1. 分析需求
回顾 discovery 阶段的输出，确认：核心目标、约束条件、成功标准。
如果 discovery 产出了产品调性（关键词标签集），将其作为后续技术选型的约束输入。

### 2. 探索方案
- 现有技术栈和资源
- 可选实现方式及优劣
- **查询依赖文档**：优先 context7，备用 WebSearch，关注 API 变更和最佳实践

### 3. 技术栈选型（涉及前端时必做）

针对项目中每个主要功能区域，研究可选的技术方案并让用户做选择。

1. **列出决策点**：哪些地方需要选技术？（如：画布用什么库、状态管理用什么、构建工具用什么）
2. **逐个研究**：每个决策点搜索 2-3 个候选方案，列出优劣（用 context7 查文档、WebSearch 查社区评价）
3. **频繁确认**：每个决策点都用 AskUserQuestion 让用户选择，不要攒到最后一起问
4. **记录结论**：每个决策点的选择结果和理由，作为后续模块拆分的输入

### 4. 全链路约束探测

**在模块拆分之前，必须探测并确认整条链路上的所有环境约束。** 目的是提前发现阻塞点，避免执行阶段才撞墙。

探测清单（按需选取）：

| 类别 | 探测项 | 方法 |
|------|--------|------|
| 本地环境 | 数据库是否可连接、密码是否正确 | 尝试连接，失败则立即询问用户 |
| 本地环境 | 必要 CLI 工具是否在 PATH 中 | `which`/`where` 检查 |
| 密钥/凭证 | API Key 是否有效、额度是否充足 | 调用一次轻量接口验证 |
| 远程环境 | VPS 是否可 SSH 连接 | `ssh -o ConnectTimeout=5` 测试 |
| 远程环境 | 服务器上必要软件版本 | SSH 执行 `node -v` / `mysql --version` 等 |
| 域名/DNS | 域名是否已解析到目标 IP | `nslookup` / `dig` 检查 |
| Git/CI | GitHub 仓库是否存在、gh CLI 是否已认证 | `gh auth status` + `git remote -v` |
| 部署工具 | deploy-setup 是否可用 | `deploy-setup --version` 或备用路径 |

**探测结果处理**：
- 全部通过 → 记录到规划文档的"部署和运维"部分，继续模块拆分
- 有失败项 → 立即用 AskUserQuestion 告知用户具体问题和解决方法，等用户解决后再继续
- 不要跳过失败项，不要假设"到时候再说"

### 5. 模块拆分与依赖排序（核心输出）

**将系统拆解为可独立开发的模块，并排出线性开发顺序。**

拆分原则：
- 每个模块有明确的职责边界
- 模块间通过接口/数据格式通信，尽量松耦合
- 基础模块（被多个模块依赖的）排在前面

每个模块必须说明：
- **模块名称**
- **职责**：这个模块做什么
- **技术选型**：用什么库/框架（引用步骤3的决策结论）
- **对外接口**：暴露什么给其他模块（API、组件 Props、事件等）
- **依赖**：依赖哪些上游模块（必须是已排在前面的模块）
- **核心处理逻辑**：伪代码/配置级别的描述
- **异常处理**：失败了怎么办

**排序输出格式**：
```
模块1: 基础设施/数据层（无依赖）
  ↓
模块2: 核心业务逻辑（依赖模块1）
  ↓
模块3: 前端组件A（依赖模块2）
  ↓
模块4: 前端组件B（依赖模块2、模块3）
  ↓
模块5: 集成与联调（依赖全部）
```

### 6. 编写项目规划文档

将以上所有成果组织为该项目的CLAUDE.md。CLAUDE.md的结构为六块结构（严格遵循，缺的章节留占位）：

#### 1. 项目概述
- 项目名称和简介
- 主要功能和目标
- 技术栈（汇总步骤3的所有选型结论）

#### 2. 架构设计
- 整体架构图（文字描述模块关系）
- 模块划分（步骤5的模块列表）
- 数据流向（模块间的接口和数据格式）
- 关键设计决策及原因（步骤3每个决策点的结论和理由）

#### 3. 开发规范
- 代码风格
- 命名规范
- 目录结构
- 提交规范

#### 4. 核心模块说明
- 每个模块的职责、对外接口、核心处理逻辑、异常处理
- 即步骤5中每个模块的详细描述

#### 5. API文档
- 接口列表（如有后端）
- 请求/响应格式
- 错误码说明

#### 6. 部署和运维
- 环境要求（步骤4探测结果）
- 部署步骤
- 配置说明
- 常见问题

### 6.5 微任务拆解

模块拆分完成后，将每个模块进一步拆解为微任务。这是 Planning 的最后一步产出，直接对接 Execution。

**微任务标准**：
- 每个任务 2-5 分钟可完成
- 每个任务包含：文件路径、变更描述、验证命令、预期输出
- 执行者只看任务描述就能完成，不需要额外的项目上下文
- 超过 5 分钟的任务继续拆分

**微任务格式**（写入 TaskCreate 时使用）：
```
subject: [简短任务名]
description: |
  文件：[精确文件路径]
  变更：[具体做什么，不是"加个函数"而是函数签名和核心逻辑]
  验证：[跑什么命令]
  预期：[命令输出应该是什么]
```

### 7. 确认方案

向用户展示完整的六块规划文档，获得确认。**必须等用户明确确认后才进入下一步。**

### 8. 规划固化（用户确认后执行）

**动作A：写入项目规划文档**
将六块结构的规划文档写入项目本地 `.claude/CLAUDE.md`。

**动作B：创建结构化任务（线性依赖链）**
按模块开发顺序，通过 TaskCreate 逐个创建任务。每个任务包含：
- 具体实现内容（对应模块的核心处理逻辑）
- 交付标准（可验证的验收条件）
- **blockedBy**：设置为上游模块的任务 ID，形成线性依赖链

示例：
```
Task 1: 数据层搭建          → blockedBy: []
Task 2: 核心业务逻辑        → blockedBy: [Task 1]
Task 3: 前端组件A           → blockedBy: [Task 2]
Task 4: 前端组件B           → blockedBy: [Task 3]
Task 5: 集成联调            → blockedBy: [Task 4]
```

## 阶段结束

规划固化完成后，用 AskUserQuestion 向用户确认下一步：

选项必须包含：
- "进入 Execution"（推荐）
- "需要修改计划"
- "暂停"

## 规划完成检查清单

- [ ] 已理解现有架构和约束
- [ ] 已查询涉及的第三方库/平台文档
- [ ] 已完成技术栈选型，每个决策点都经用户确认
- [ ] 已完成全链路约束探测，所有阻塞项已解决
- [ ] 已拆分模块并排出线性开发顺序
- [ ] 已编写完整六块结构的规划文档
- [ ] 方案细到"照着做就行"
- [ ] 确认没有撰写代码文件
- [ ] 用户已确认方案
- [ ] 已写入 .claude/CLAUDE.md
- [ ] 已通过 TaskCreate 创建所有任务（含交付标准和 blockedBy 依赖链）
