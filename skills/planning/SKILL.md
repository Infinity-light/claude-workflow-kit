## 铁律

没有文件骨架不进入 Execution。产出者和验证者必须是不同的子代理。

## 常见借口（都是错的）

| 你可能会想 | 为什么是错的 |
|-----------|------------|
| "任务太简单不需要走流程" | 简单任务走流程成本极低，不走流程出错成本极高 |
| "我自己检查一下就行" | 自己批改自己的作业永远觉得没问题。产出和验证必须分离 |
| "先写个 plan 文档再说" | plan 文档是独立于代码的第二信息源，会和代码不一致。契约直接写在文件里 |
| "先开始做，边做边补契约" | 边做边补 = 没有契约。子代理需要在开始前看到完整规格 |
| "用多个子代理分别写不同文件的契约" | 多个子代理没有共享上下文，接口会不一致。一个子代理写全部契约 |

## 核心原则

- **主 Agent 只调度，不产出具体文件内容** — 具体产出交给子代理
- **产出-验证分离** — 每个产出物由一个子代理写，另一个子代理验证
- **一个子代理写全部文件契约** — 保证跨文件接口一致性
- **契约写在文件里，不写在独立文档里** — 信息和实现共生，减少跨边界衰减
- **测试骨架只写描述和 TODO，不写断言** — 断言是 Execution RED 阶段的事

## 核心流程

```
步骤1: 读取 PRD + 技术选型（主Agent）
  ↓
步骤2: 架构设计（子代理产出 + 子代理验证 + 用户确认）
  ↓
步骤3: 创建文件骨架（一个子代理写全部契约 + 验证子代理检查一致性）
  ↓
步骤4: 创建测试骨架（子代理产出 + 验证）
  ↓
步骤5: 用户确认 → 更新 CLAUDE.md → 进入 Execution
```

---

### 步骤1: 读取 PRD + 技术选型

**执行者**：主Agent 自己做。

1. **读取 Discovery 产出**：从 `.claude/discovery/PRD.md` 读取需求文档
2. **读取探索性制品**：从 `.claude/exploration/` 读取用户选中的方案（粗糙原型等）
3. **读取项目 CLAUDE.md**：了解已有的技术栈和架构（如果是已有项目）
4. **技术选型**（新项目或新技术决策时）：
   - 列出决策点
   - 逐个研究 2-3 个候选方案（优先 context7，备用 WebSearch）
   - **每个决策点单独用 AskUserQuestion 让用户选择**
   - 记录所有选型结论
5. **全链路约束探测**：按需检查本地环境、密钥、远程服务器等

**不需要子代理验证** — 这步的产出是用户的选择和环境确认。

---

### 步骤2: 架构设计

**执行者**：子代理A（产出）+ 子代理B（验证）

1. **启动产出子代理**：
   - 传入：PRD 文件路径 + 探索性制品路径 + 技术选型结论
   - 指令：设计整体架构，产出内容包括：
     - 目录结构（精确到每个文件）
     - 模块划分和模块间依赖关系
     - 数据流向
     - 环境变量清单
     - 前后端通信约定（如适用）
     - 数据库设计（如适用）

2. **启动验证子代理**：
   - 传入：产出子代理的结果
   - 指令：读取 `references/architecture-checklist.md`，逐项验证
   - 不通过 → 修补 → 再验证

3. **主Agent 向用户展示架构设计，用 AskUserQuestion 确认**

---

### 步骤3: 创建文件骨架（核心步骤）

**执行者**：一个子代理产出全部文件 + 另一个子代理验证

1. **启动产出子代理**（一个子代理完成所有文件）：
   - 传入：PRD 文件路径 + 架构设计 + 探索性制品路径 + 技术选型
   - 指令：根据架构设计，创建项目中所有实际代码文件
   - 每个文件只有头部契约注释，没有实现代码
   - 契约使用 YAML frontmatter 格式，在对应语言的注释块内：

   **TypeScript/JavaScript 格式**：
   ```typescript
   /**
    * ---
    * role: [文件职责一句话描述]
    * depends: [依赖文件路径列表]
    * status: PENDING
    * functions:
    *   - functionName(param: type): ReturnType
    *     行为描述，包括边界条件和错误处理
    * ---
    */
   ```

   **Python 格式**：
   ```python
   """
   ---
   role: [文件职责一句话描述]
   depends: [依赖文件路径列表]
   status: PENDING
   functions:
     - function_name(param: type) -> ReturnType
       行为描述，包括边界条件和错误处理
   ---
   """
   ```

   **一个子代理写全部文件的原因**：保证跨文件接口一致性。如果 A 文件调用 B 文件的函数，签名必须匹配。

2. **启动验证子代理**：
   - 传入：所有已创建文件的路径列表
   - 指令：
     - 逐文件检查契约完整性（role、depends、status、functions 是否齐全）
     - 检查跨文件接口一致性（A 文件 depends B，B 文件是否暴露了 A 需要的函数）
     - 检查依赖关系是否有环
     - 不通过 → 列出问题 → 新子代理修补 → 再验证

---

### 步骤4: 创建测试骨架

**执行者**：子代理产出 + 子代理验证

1. **启动产出子代理**：
   - 传入：所有文件骨架的路径列表 + PRD 中的验收标准
   - 指令：为每个有 functions 的文件创建对应的测试文件
   - 测试文件只有 describe/test 块和描述文字，断言部分写 `// TODO: implement assertion`
   - 测试用例来源于 PRD 的验收标准和文件契约的行为描述

2. **启动验证子代理**：
   - 检查每个有契约的文件是否都有对应的测试文件
   - 检查测试用例是否覆盖了契约中描述的所有行为和边界条件
   - 不通过 → 修补 → 再验证

---

### 步骤5: 固化

1. **用 AskUserQuestion 让用户确认整体骨架**
2. **更新项目 CLAUDE.md**：通过子代理将本轮新增的架构信息写入 CLAUDE.md
   - 新增的模块和职责
   - 新增的技术选型决策
   - 目录结构
3. **运行进度扫描脚本**：`python -X utf8 .claude/scripts/scan_contracts.py <项目根目录>`，生成初始 `.claude/progress.md`

---

## 阶段结束

固化完成后，用 AskUserQuestion 向用户确认下一步：

选项必须包含：
- "进入 Execution"（推荐）
- "需要修改骨架"
- "暂停"
