## 铁律

没有通过规格审查的任务不算完成。主Agent 只调度，不直接执行。

## 常见借口（都是错的）

| 你可能会想 | 为什么是错的 |
|-----------|------------|
| "这个任务太简单，我直接做了" | 主Agent 直接做 = 跳过审查。再简单也要走子代理 |
| "规格审查太慢，先跳过" | 没审查的代码是薛定谔的代码——你不知道它对不对 |
| "子代理上下文够了，不需要传完整任务文本" | 子代理没有你的上下文。传引用 = 传空气 |
| "Worktree 太麻烦，直接在主分支改" | 3个以上任务在主分支改，搞砸了没有回头路 |

## 核心职责

Execution 是通用调度层，不绑定任何特定领域。
具体怎么做由 workflow reference 决定，领域约束由 practices 提供。

## SDD 编排（Subagent-Driven Development）

### 0. 环境准备

检查 TaskList 中 pending 任务数量：
- 超过 3 个 → 创建 Git Worktree 隔离分支，确认基线测试通过
- 3 个及以下 → 在当前分支工作

### 1. 领取任务

从 TaskList 中按 ID 顺序领取 pending 任务：
- TaskGet 读取任务详情（含交付标准）
- TaskUpdate 标记为 in_progress

**不要自行拆分任务。任务已在 planning 阶段创建。**

### 2. 匹配 Workflow

根据任务内容，到 `workflows/` 目录下探索，找到最匹配的 workflow reference：
- 用 Glob 扫描 `workflows/` 下的文件列表
- 根据文件名和任务类型判断匹配度
- 读取匹配的 workflow，按其中的流程指引执行

如果没有匹配的 workflow，按通用逻辑执行。

### 3. 调度实现子代理

启动**全新子代理**执行任务：
- 将任务的**完整文本**传入 prompt（从 TaskGet 获取的 description 全文，不是引用）
- 指定需要读取的 workflow 和 practices 文件路径
- 子代理按 workflow 流程执行并报告结果

**约束传递**：子代理不会继承主Agent 上下文，必须在 prompt 中显式包含所有必要信息。

### 4. 规格审查

实现完成后，启动**另一个全新子代理**做规格审查：
- 传入任务的完整描述（同步骤3）
- 传入实现者的报告
- 审查指令："实现者的报告可能不完整，你必须独立验证：每个要求是否被满足、是否做了不该做的事"
- 审查不通过 → 启动新子代理修复 → 重新审查
- 审查通过 → 进入步骤 5

### 5. 管理任务状态

- 审查通过 → TaskUpdate 标记为 completed → 回到步骤 1 领取下一个
- 执行中遇到错误或阻塞 → 转入 diagnosis 阶段

### 6. 全部完成

所有任务完成后：
- 如果创建了 Worktree → 用 AskUserQuestion 提供分支处理选项：合并到主分支 / 创建 PR / 保留分支 / 丢弃，然后清理 Worktree
- 简要告知用户结果

## 执行中遇到不确定情况

1. 暂停执行
2. 进行必要的探索（查代码、查文档、查 references/）
3. 确认后继续

## 阶段结束

所有任务完成后，用 AskUserQuestion 向用户确认下一步：

选项必须包含：
- "进入 Documentation-update"（推荐）
- "需要补充任务"
- "暂停"
